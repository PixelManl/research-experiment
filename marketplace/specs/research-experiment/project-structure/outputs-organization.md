# Outputs Organization Contract

Outputs must be traceable, indexable, and invalidatable. The goal is not to keep every artifact forever; the goal is to know what each artifact means — from files, in one command.

## When to Read

Read this before:

- creating a formal run output directory;
- writing metrics, figures, diagnostics, artifacts, or processed data;
- locating, comparing, citing, cleaning, or archiving outputs;
- changing run validity (promote, supersede, invalidate).

## Layer model

```text
outputs/
├── index.md                          # generated cross-slot view — never hand-edited
└── <task-slot>/
    ├── runs.jsonl                    # REGISTRY: append-only event log, the truth
    ├── index.md                      # generated slot view — never hand-edited
    └── <YYYY-MM-DD>/<HHMMSS>-<run-name>/   # run directory: full detail
```

The registry is the machine-readable truth for run identity, status, and lineage; see [../experiment-runtime/run-registry.md](../experiment-runtime/run-registry.md). Markdown indexes are views regenerated by `runs.py render`. Run directories hold the complete evidence.

Avoid `MM:dd` in paths; `:` is invalid on some filesystems.

## Run identity

Every formal run has a registry id `<task-slot>#<seq>` (e.g. `ppo-handwritten#0007`). Cite runs by id — in ledgers, figures, reports, and chat. Paths are storage; ids are references.

## Required run directory contents

```text
run.log
status.json          # execution axis only: running|success|failed
metrics.json
provenance.json      # identity card: ids, git, config hash, command, host
git.diff.patch       # human-browsable view of uncommitted changes
command.txt
environment.txt
.hydra/
artifacts/
figures/
diagnostics/
```

`provenance.json` fields are defined in [../experiment-runtime/provenance.md](../experiment-runtime/provenance.md). It contains no validity field.

## Two status axes

Execution (`status.json`, written by the process):

- `running` / `success` / `failed` — "did it finish".

Validity (registry events only, via `runs.py promote|invalidate`):

- `exploratory`: default at birth; may inform debugging, not claims.
- `valid`: passed checks; can support internal decisions. Requires a resolvable git snapshot.
- `claim-ready`: human approved; can support paper/report claims. Requires pushed snapshot/commit and `--by <human>`.
- `invalidated`: must not support future claims.
- `superseded`: replaced by a later valid run.

A run can be `success` + `invalidated`, or `failed` + kept as failure evidence. Never store validity in `status.json` or `provenance.json`; the registry is its only home.

## Canonical run

Per task-slot, at most one run is canonical (`role: canonical`, validity `valid`/`claim-ready`). `runs.py latest <slot>` returns it or reports vacancy. Promoting a new canonical requires superseding the old one or declaring a non-canonical role (`replicate`/`ablation`). See [../experiment-runtime/run-registry.md](../experiment-runtime/run-registry.md).

## Locating and comparing runs

Use the CLI, not directory walks or one-off scripts:

```bash
python scripts/common/runs.py latest ppo-handwritten
python scripts/common/runs.py list ppo-handwritten --validity valid --since 2026-05-28
python scripts/common/runs.py show "ppo-handwritten#0007"
python scripts/common/runs.py compare "ppo-handwritten#0004" "ppo-handwritten#0007"
```

`compare` answers "two runs, three days apart, both look real — what changed?" with the code diff (from snapshots, so dirty runs compare exactly), key-level config diff, and metric deltas.

## Data processing outputs

Persistent processed data belongs in:

```text
data/processed/<task-slot>/<dataset-version>/
```

A data processing script under `scripts/<task-slot>/process_data.py` must write:

```text
manifest.json
input_hashes.json
config.yaml
```

Run-specific derived artifacts remain under `outputs/<task-slot>/.../artifacts/`.

## Cleaning and archiving

- Heavy artifacts of `superseded`/`invalidated` runs may be deleted; their registry lines and run directories' provenance files stay.
- Never delete `runs.jsonl` lines; the log is append-only history.
- Before archiving a task-slot, run `runs.py check <slot>` and resolve errors.

## Git ignore

Track the registry and generated indexes, ignore heavy outputs. This is also a snapshot-correctness precondition (snapshots capture everything not ignored):

```gitignore
outputs/**
!outputs/
!outputs/index.md
!outputs/*/
!outputs/*/runs.jsonl
!outputs/*/index.md
```

Do not rely on ignored outputs as the only record of scientific conclusions; important conclusions belong in `docs/research-log/`.

## Forbidden

- Hand-editing `runs.jsonl` or any `outputs/**/index.md`.
- Walking date directories or writing ad hoc aggregation scripts for queries `runs.py` answers.
- Citing runs by directory path instead of run id.
- Marking validity anywhere except registry events.
- Two coexisting "current" results without a canonical/supersede decision.
